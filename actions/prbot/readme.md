# PR Bot

Create or re-use a **pull request** in a repository. PRs are always opened **in the repo you specify** (same repo as the workflow, or another repo). Zero-dependency (Python stdlib only). Idempotent: if a PR already exists for that head/base, it skips creation. **No fork workflow** — use a single management repo that opens PRs in target repos.

---

## What it does

- **Target repo**: You set `repo` to where the PR should be created (current repo or another repo).
- **Checks** for an existing open PR with the same head and base (paginated).
- **Creates** a new PR only when none exists; appends a footer to the body: `Generated by Automation | ID: [project-prbot-{timestamp}]`.
- **Logs** with `[PROJECT-PRBOT]` and debug codes; 4xx errors show a clear reason.

---

## Inputs

| Input | Required | Default | Description |
|-------|----------|---------|-------------|
| `repo` | Yes | `github.repository` | Repo where the PR is created: `owner/repo` or repo name. |
| `title` | Yes | — | PR title. |
| `source_branch` | Yes | — | Branch name for the PR head (branch must exist in the target repo). |
| `target_branch` | No | `main` | Base branch in the target repo. |
| `payload` | No | `''` | Optional PR body; footer is always appended. |
| `github_token` | Yes | `github.token` | Token with write access to the target repo. |

---

## Scenarios

### 1. PR in the same repo (where the workflow runs)

Workflow runs in repo **R**. Open a PR in **R** from branch `feature-x` into `main`:

```yaml
- uses: ./.github/actions/prbot
  with:
    repo: ${{ github.repository }}
    title: "Feature X"
    source_branch: feature-x
    target_branch: main
    github_token: ${{ secrets.GITHUB_TOKEN }}
```

### 2. Management repo: open a PR in another repo

You have a **single management repo** (e.g. `my-org/platform` or `my-org/automation`) that pushes code to branches in **other repos**. You want to **raise a PR in that other repo**, not in the management repo.

- Workflow runs in: **management repo** (e.g. `my-org/platform`).
- It has already pushed (or will push) a branch to the **target repo** (e.g. `my-org/docs-site`).
- You want the PR to be **created in the target repo** (docs-site), from that branch into `main`.

Set `repo` to the **target repo** where the PR should appear. Use the **branch name only** for `source_branch` (that branch must exist in the target repo). The token must have push/PR access to the target repo.

```yaml
# In management repo workflow (e.g. my-org/platform)
- name: Push to target repo (your existing step)
  run: |
    git push https://x-access-token:${{ secrets.GITHUB_TOKEN }}@github.com/my-org/docs-site.git HEAD:docs-sync

- name: Open PR in target repo
  uses: my-org/platform/actions/prbot@main
  with:
    repo: my-org/docs-site
    title: "Sync docs from platform"
    source_branch: docs-sync
    target_branch: main
    github_token: ${{ secrets.GITHUB_TOKEN }}
```

So: **PR is always created in the repo you pass as `repo`.** With a management repo, that repo is the **target** repo (e.g. docs-site), and both branches (e.g. `docs-sync` and `main`) live there.

### 3. Custom PR body

```yaml
- uses: ./.github/actions/prbot
  with:
    repo: ${{ github.repository }}
    title: "Release notes"
    source_branch: release-1.0
    target_branch: main
    payload: |
      ## Summary
      - New feature A
      - Fix for B
    github_token: ${{ secrets.GITHUB_TOKEN }}
```

### 4. Idempotent (e.g. nightly sync)

Re-running with the same inputs does not create a second PR; it finds the existing one and exits successfully.

```yaml
- uses: ./.github/actions/prbot
  with:
    repo: ${{ github.repository }}
    title: "Nightly sync"
    source_branch: nightly-sync
    target_branch: main
    github_token: ${{ secrets.GITHUB_TOKEN }}
```

---

## Requirements

- **Python 3** (stdlib only).
- **Token**: `GITHUB_TOKEN` or a PAT with `repo` scope; for opening PRs in another repo, the token must have access to that repo.
